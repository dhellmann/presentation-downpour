Downpour:
Moving Workloads with Ansible

Ansible Open Source Day, May, 2017

Doug Hellmann, Red Hat

Our Requirements

  • Download all tenant data from one cloud
  • Upload it to another cloud
  • Retain relationships
  • Support multiple OpenStack versions
  • Be flexible
  • Moving public/private
  • Moving between public cloud providers
  • Upgrades

Range of Use Cases

Easy Hard
Ownership Admin Tenant
Backend Shared storage Shared nothing
Applications One per tenant Rats nest

Shared storage: optimizations for moving block data

Downpour focuses on one end of that spectrum.

Tenant ACLs

No admin credentials are needed.

Shared Nothing

Works on completely disconnected clouds.

Multi-application Tenant

Export a subset of a tenant's resources.

Trade-offs

  • Not a live migration tool.
  • The tool does not prevent application disruption.
  • Requires full data export.
  • UUIDs are not preserved.

Multi-phase approach

  • Select resources
  • Export
  • Import
  • Cleanup old resources

At each phase there is an opportunity to stop and revise the inputs and outputs manually.

Select Resources

  • downpour query
  • Regular expressions
  • Flavor
  • Network?
  • Suggestions?

Phase I is to build up the set of resources to be exported.

This is where we untangle the rats nest in a tenant running several distinct applications.

Export

  • downpour export
  • Explicitly listed resources (image, volume, server)
  • Automatic dependency resolution

Phase II exports the data listed in the control file.

Pull in dependencies like networks, volumes, and images automatically.

Export

# Resources to be exported by downpour
servers:
  - name: "downpour-demo-tiny"

This is an example input file for the export phase. It would normally be produced by running a query operation to find resources and spell them out explicitly here.

Import

  • Spoiler alert: ansible-playbook
  • Idempotent
  • Someone else wrote it!

Phase III imports the data into the new cloud.

This is where Ansible comes in.

Import

  tasks:
  - lineinfile:
      create: 'yes'
      dest: uuids.csv
      insertbefore: BOF
      line: '"Resource Type","Resource Name","Old","New"'
      regexp: '"Resource Type","Resource Name","Old","New"'
      state: present
    name: Initializing UUID mapping file

This part of is the playbook generated for the server in the demo.

The first step is to create a file to track the UUID changes.

Import

  - name: Add security group downpour-demo
    os_security_group:
      description: Demo group used for downpour
      name: downpour-demo
      state: present
    register: sg001
  - lineinfile:
      dest: uuids.csv
      insertafter: EOF
      line: '"security group","downpour-demo","6deea469-54bd-4846-b12a
      -79fa6b482280","{{sg001.secgroup.id}}"'
      regexp: '"security group","downpour-demo","6deea469-54bd-4846-b1
      2a-79fa6b482280"'
      state: present
    name: Map UUID for downpour-demo

Dependencies such as the security groups used by the server are automatically included in the playbook without needing to be specified.

Then the CSV file is updated with the line mapping the name and old UUID to the new value.

Of course, the security group rules are included, too.

Import

  - name: Create volume downpour-demo-tiny
    os_volume:
      display_description: null
      display_name: downpour-demo-tiny
      size: 1
      state: present
    register: vol002

Any volumes attached to the instance are also recreated.

For now an empty volume is created. I'm still working on the data upload.
